CHR Workbench in 60 seconds

What you're looking at. Every NES cartridge stores its art as a grid of 8×8-pixel tiles called CHR data. The sheet on the left is all of it. Sprites, text, backgrounds — if you can see it in the game, it's made of these tiles.

Pattern tables. The console reads tiles in pages of 256, labelled PT0, PT1… on the sheet. When game code (or an AI assistant) refers to “tile $A3”, it means the position within a pattern table — the status bar shows both numbers for whatever you select.

Why only 4 shades? Tiles don't store colors, just slots 0–3. The game assigns real colors at runtime, and reuses the same tile with different palettes. The preview palettes here are cosmetic — your edits always save as slots.

8×16 pairs. Many games build characters from two stacked tiles. If a sprite sheet looks scrambled, toggle the pair view.

Suggested workflow. 1) Redraw tiles here. 2) Download the patched .nes and test it in an emulator (Mesen, FCEUX). 3) Hit Copy notes and paste into your AI coding assistant — it lists exactly which tiles you changed, with the ids and file offsets code work needs (palette swaps, tile remapping, level edits).

Heads-up. Games whose header reports no CHR-ROM draw with CHR-RAM — their art is generated by code and can't be edited here. And bank-switched games reuse tiles in several places, so an edit can show up in more than one screen.
