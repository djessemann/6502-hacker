Help

What you're looking at

NES art is stored as 8×8-pixel tiles called CHR data. The sheet on the left is every tile in the file — sprites, text, backgrounds. Click a tile to edit it on the right.

Pattern tables

The console reads tiles in pages of 256, labelled PT0, PT1… on the sheet. Game code refers to a tile by its position within a page — “tile $A3” — so the status bar shows that id and the tile's file offset alongside the overall index.

Colors

Tiles store slots 0–3, not colors. The game maps slots to real colors while it runs, and often reuses one tile with several palettes. The preview palettes only change how this editor displays slots — saved data is identical in every preview.

8×16 pairs

Many games build characters from two stacked tiles. If a sprite sheet looks interleaved, turn on the pair view: the sheet reorders into 8×16 units and the editor edits both tiles together — flips included.

Assembly bench

Tiles are only ingredients — each game's own data decides how they compose into a character, and no tool can read that generically. The bench lets you rebuild one by eye: tap a bench cell, then tap tiles in the sheet to place them (placement advances to the next empty cell). Flip or remove the selected cell with the buttons. The bench redraws from live data, so editing a placed tile updates the assembled character as you paint. Copy as JSON exports the bench in the layout format below.

Layouts

For pixel-accurate in-game characters, import layout JSON via Import… in the Layouts panel. The dialog's Copy prompt button gives your AI coding assistant exact instructions for extracting a game's metasprite data into the right format. Imported sprites render live from the working tiles — click anywhere on one to jump to that tile in the editor. Layouts live in memory only; keep the JSON with your project and re-paste next session.

CHR-RAM games

Games with no CHR-ROM (Zelda, Metroid, Final Fantasy…) keep their art inside PRG-ROM and copy it to CHR-RAM while running. These now load as the whole PRG-ROM decoded as tiles: program code looks like noise, and uncompressed graphics visibly pop out as you scroll. Graphics aren't always 16-byte aligned — use the offs −/+ control to shift the window until tiles snap into focus. Careful: painting over noise edits program code and will break the game. Compressed graphics stay noise; unpacking them is per-game work.

Workflow

  1. Redraw tiles. Edited tiles get a red corner mark.
  2. ↓ .nes downloads a patched copy of the ROM — only the art bytes change. Test it in an emulator such as Mesen or FCEUX.
  3. Copy notes puts a summary of your edits on the clipboard — mapper, tile ids, file offsets — ready to paste into an AI coding assistant for the code side of the hack: palette swaps, tile remapping, level edits.

Limitations

Bank-switched games reuse tiles, so one edit can appear on several screens. Everything stays in your browser — closing the tab discards unexported work.
